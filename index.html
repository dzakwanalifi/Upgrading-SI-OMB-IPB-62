Clarity (Kejelasan)

"Paham dulu masalahnya apa!"

‚ùì Tanya: "Maksudnya?", "Contoh?", "Goal?"

‚ùå Web Lambat?

‚ûî

‚úÖ Pas Apa? Dimana?

Kunci: JELASKAN DULU!

Clarity! Paling dasar. Paham nggak sih kita sama masalahnya? Pertanyaannya: 'Apa *sebenarnya* yang dimaksud?', 'Bisa kasih contoh spesifik?'. Misal laporan web lambat. Harus jelas: 'Lambatnya pas ngapain? Di halaman mana? Sejak kapan?' [Jelaskan potensi solusi beda berdasarkan kejelasan masalah: optimasi query vs upgrade server vs queueing]. Kejelasan di awal krusial. Contoh Outsco: Sebelum bikin sistem, aku tanya dulu: 'Butuhnya apa aja *detailnya*?'

Accuracy (Ketepatan)

"Info/Datanya beneran valid nggak?"

‚ùì Tanya: "Sumber?", "Validasi?", "Update?"

‚ùì Data GForm Maba

‚ûî

‚úÖ Cross-check? Valid?

Ingat Prinsip GIGO! (Garbage In, Garbage Out)

Accuracy! Info yang didapet bener nggak? Valid nggak? Sumbernya? Bisa diverifikasi? Update? Misal, data Maba di GForm. Yakin semua diisi bener? Perlu *cross-check* data resmi? Ingat GIGO: *Garbage In, Garbage Out*. Kalau data inputnya nggak akurat, hasilnya bakal ngaco. Pengalaman lomba: Data mentah *wajib* dicek dulu keakuratannya.

Relevance (Keterkaitan)

"Nyambung nggak sih infonya sama masalah utama?"

‚ùì Tanya: "Hubungannya?", "Penting buat solusi?"

Lagi Debugging Data Maba Error:

Info Merk Laptop Panitia VS Log Server & Query SQL

Kunci: FOKUS YANG PENTING!

Relevance! Info atau idenya nyambung sama masalah yang dibahas nggak? Penting nggak buat solusi? Hubungannya apa? Misal, pas *debugging* data Maba nggak masuk database. Info merk laptop panitia nggak relevan. Tapi info log server, versi bahasa/DB, struktur tabel itu sangat relevan. Fokus ke yang penting biar nggak buang waktu.

Depth (Kedalaman)

"Udah gali sampe akarnya belum?"

‚ùì Tanya: "Kenapa?", "Kenapa lagi?", "Faktor lain?"

Gejala: Data Nggak Sinkron

Akar Masalah (Mungkin):
- Script Buggy?
- API Change?
- Setting Salah?
- Koneksi Intermittent?

Kunci: GALI AKARNYA! (Kayak 5 Whys)

Depth! Udah liat sampe akar masalahnya belum, atau cuma di permukaan? Faktor apa yang bikin kompleks? Ada asumsi tersembunyi? Misal, data kehadiran nggak sinkron. Jangan cuma bilang 'koneksi jelek'. Tanya lagi: '*Kenapa* koneksinya jelek? Atau *script*-nya yg *buggy*? Atau API-nya berubah?' Terus gali pake 'kenapa' (Kayak 5 Whys). Pengalaman *debugging* apps: Nggak cuma liat error di FE. Tapi runut ke BE, cek query ke DB. Perlu liat *end-to-end*, jangan cuma 'nambal' gejala.

Kunci: ARGUMEN KUAT!

Logic! Alur mikirnya masuk akal nggak? Kesimpulannya didukung bukti kuat? *Make sense* nggak? Ada lompatan logika? Misal, kasus bug *race condition*. Hipotesisnya karena akses barengan. Solusi: implementasi *locking*. Logis? Ya, karena *locking* memang solusi standar untuk itu. Pengalaman bikin sistem Outsco: Alur dari Input (GForm) -> Proses (Sheet) -> Output (Dashboard/Report) harus logis & bisa dijelasin 'kenapa'-nya. Setiap langkah harus nyambung.

Kenali Jebakan Pikiran (Bias) üòµ

Otak kita suka 'jalan pintas', tapi kadang menyesatkan!

Sekarang kita masuk ke 'jebakan batman' dalam mikir: Cognitive Bias. Otak kita suka jalan pintas, tapi kadang menyesatkan. Penting dikenali biar keputusan kita nggak keliru, terutama pas dikejar waktu. Yuk kita lihat apa aja jebakannya... [Panah bawah]

Apa Sih Bias Kognitif Itu?

"Jalan Pintas" mental otomatis yang otak kita pakai buat ambil keputusan cepat.

Masalahnya: Jalan pintas ini nggak selalu akurat & bisa bikin salah nilai!

Jadi gampangnya, bias kognitif itu kayak 'shortcut' di otak kita. Biar nggak lama mikir. Masalahnya: shortcut ini seringkali nggak liat konteks atau data lengkap, jadi hasilnya bisa salah. Kayak ngambil jalan pintas tapi malah nyasar. Kenali biar kita lebih waspada.

Contoh Jebakan Pikiran yang Sering Muncul:

Bias Konfirmasi

Cari info yg cocok sama
pendapat awal aja.

(Misal: Yakin error di DB, cuma cek itu).

Jumping to Conclusions

Langsung nyimpulin
padahal bukti minim.

(Misal: 1 laporan error -> "Sistem down!").

Groupthink

Ikut mayoritas
biar 'aman' / gak debat.

(Misal: Semua setuju A, padahal ragu).

Ini beberapa contoh yang sering kejadian: [Bias Konfirmasi]: Cenderung cari bukti yang dukung apa yang udah kita percaya aja. Misal: Yakin error di DB, cuma cek itu. [Jumping to Conclusions]: Langsung ambil kesimpulan padahal infonya dikit. Misal: 1 laporan error -> "Sistem down!". [Groupthink]: Ikut-ikutan aja sama mayoritas biar 'aman'. Misal: Semua setuju A, padahal ragu. Pasti pernah ngalamin kan? Nah, gimana cara ngelawannya? [Next]
